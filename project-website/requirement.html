Requirements

Project Background

The RCGP "ARI NLP Tool" is an open source NLP engine for facilitating and improving interactive services with GP call queues. It aims to provide users with a more simplified and efficient solution for booking appointment with GP surgery by introducing the interactive voice response system.

The IVR system would take users requests by analysing their answers to pre-set questions, then direct them to the corresponding place. It significantly improves users' experience and help reduces the current massive workload of front-desk receptionsts at GP practice.

Requirement gathering

To better understand the requirements from our clients, we set up seperate meetings with each of the client from both NHS and RCGP in the first few weeks. During the meetings, we discussed about the potential application and the exact features that are required in the final project. After some further clarification about ambiguities, we were able to compile a MoSCoW list over the course of the meetings as requested.

Some of the questions we asked during the interview include:

  • What the existing telephoney system that's currently used within the GP practice?
  • What operation system should our final solution expected to be run on? Window, Linus, Ubuntu?
  • Is there any regulation we should be aware of since the project consists handling patients' information?

    • Apart from interviewing with clients, we also gathered some insights from the "patients" via anonymous questionnaires. Due to the current lockdown situation, we decided to conduct the research online by asking fellow students to fill in the Google form we provided and analyse their answers afterwards. This method was suggested by Dr Chris Evans, the HCI lecturer of the module.

    Client Interviews

    Below are the snippets we recorded from our interviews with GP and experts from NHS and RCGP

    • Dr Elisabeth Krymalowski (RCGP)

      Q: Is there any existing project that is currently in used that's similar to our solution?

      A: Depending on the GP surgery, at the moment most appointments are made by the online booking system. But most likely our receptionists will have to call back and double check the exact symptoms with each patient.

    • Dr Stephanie Coughlin (RCGP)

      Q: Despite the general questions, is there anything else that we need to know from the patient?

      A: Yes, some additional questions should asked when necessary. Take back pain for an example, it's important to ask the patient's age as a subsequent question. Especially for elderly patients, back pain is considered to be a red flag which might indicate some more severe problems.

    • Dr Joseph Connor (NHS)

      Q: Is there any regulation we should be aware of since the project consists handling patients' information?

      A: Yes, due to the GDPR regulation, it is necessary that the data processing is conducted offline - meaning you need to prevent the use of cloud solution.

    User Cases

    Following our interview with the clients and the answers we gathered from online questionnaires, we were able to create some potential use cases of our project as shown below:

    • John Smith

      John is a 40-year-old male who has long been suffering for his back pain.

      Due to this reoccuring issue, John has been using the current NHS phone system to book appointments with a physio for a long time. John wants to book the appointment in one go rather than going back and forth and explain his symptoms every time.

      With the "ARI NLP tool", John is now directed to an IVR system when calling the local GP number. Instead of waiting in the massive call queue, he found himself successfully redirected to the physio after answering a few questions.

    • Sarah Jones

      Sarah is a 31-year-old receptionist who works at a local GP surgery. Her responsibilities include directing patients to the right services. However, the online booking system does not represent patient’s symptoms clearly. Additionally, due to the pandemic, there has been a surge in incoming calls requesting COVID test, which causes massive workload to redirect caLLS.

      With the "ARI NLP tool", the booking process is now fully automatic. Sarah only needs to log in the system with credentials and configure the numbers that patients will be directed to. The setting process of IVR is also very simple, which can easily done by drop-and-drop the Voiceflow file.

      Additionally, Sarah found the statistical analysis feature very useful, as it provides a direct visualisation of how many incoming calls were answered today, as well as other status including not answered/user busy or call failed.

    MoSCoW list

    Following our interview with clients and potential interviews, we gathered all requirements and compiled a MoSCoW list for out project.

    Must have:


    • A working IVR that asks questions that require a “Yes” or “No” answer. It gives a triage but does not necessarily redirect the caller for any GP (as this depends on the specific telephony system the GP has).
    • Anyone should be able to call the IVR (will require the GP to have a SIP provider).
    • Has to be able to sit in front of a Mitel System. (Specifically for the Mitel System, it must be able to redirect callers to the Mitel telephony system or a specific telephone number).
    • We must be able to get a diagram from Voiceflow using the client’s Voiceflow credentials and return the diagram in a JSON format.
    • We must have software developed to turn this Voiceflow diagram JSON format into an IVR automatically.
    • We must show on the UI whether the total number of calls into the IVR is equal to the total number of calls being redirected at the end of the IVR.
    • There must be an input on the UI to allow a user to input their Voiceflow email, password, workspace name and project name.
    • There must be an input on the UI to change the telephone number (specifically as a must have the specific internal telephone numbers for each of the IVR end results).

    Should have:


    • The only part of the program that will be allowed to access the internet is when retrieving the Voiceflow diagram. Other than this the program should be fully independent of the internet/ cloud services.
    • UI should show the call date, total number of calls and the number of calls going to each end point of the IVR. This will be displayed on the computer where the IVR sits.
    • There should be the option for the receptionist at the GP surgery (or someone else) to input their own recorded messages for the IVR (rather than using the automatic text to speech ones). They will record the messages elsewhere and then using a particular file type (such as .MP3 or .WAV) they will be able to add these in place of the automatic messages.
    • A section of the UI which allows the user to change the telephony system in place as well as relevant redirecting numbers. This will allow GP surgeries with a variety (if not all) telephony systems to use our IVR system easily through the UI settings.
    • A section of the UI should allow us to see specifically why calls were cut off during the IVR if they were. We should be able to specifically see whether the user hung-up during the IVR or if they didn’t get to the end for another reason (such as connection issues).

    Could have:


    • Information from IVR sent to relevant people in the surgery (in the form of reports to the person who the patient is being redirected to).
    • More sophisticated statistical analysis of the data in order to help the GP surgery know where most patients are going through to, the times of day etc. Collecting this into nice formats and reports with graphs that can easily be analysed by someone working at the surgery.
    • Being able to change the JSON file on the Linux machine running the IVR remotely. This would allow a receptionist to much more easily change the IVR.
    • Being able to change the telephony settings of the system remotely.
    • Being able to review reports and statistics of the system remotely.

    Won't have:


    • Not having Alexa integration.
    • Won’t have a solution and UI to build your own IVR (we’ll just be translating from Voiceflow).
    • Won’t have our own PBX or IVR solution (we’ll be most likely using Asterisk).
    • Won’t have our own text to speech model (using external library).
    • Won’t have our own speech to text model (again, using external library).
    • Won’t have questions that require more complicated answers (only “Yes” or “No”).
