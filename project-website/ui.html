UI Design

UI Design

The UI was not an extremely important part of our project. However, it was still important that a user could easily make changes to the IVR system.

We wanted to make the UI as simple as possible for a user to navigate through. For this reason, we made a simple UI, with very clearly labelled options for each aspect of the system. This took a few iterations to get right based on our client and TA feedback.

The User Interface was developed using the Python Library PyQt5.

Overall Process

Initially we had an extremely simplistic UI. No explanation of each page was present on the UI. We initially thought this was okay because we had a user manual to explain what each page was used for. When showing this to our client, he advised that some explanation of each page would be useful for a first-time user. From this feedback, we added a page information summary at the top of each page in the UI. We also added tooltips to almost every widget in the UI. This means that if a user hovers over a button or icon, they will get more information about what the button or widget does and any consequences. We then showed this to our client, and he seemed very happy with the changes, as it made the system a lot clearer to someone who wasn’t so familiar. On our first iteration of the UI, our TA advised that we should add icons to each of the buttons and that it was unclear what an IVR was. We then decided to add icons to each of the button, we went with a theme of blue for the icons to keep consistency. Blue is also the colour of the NHS logo, so this proved favourable to our clients as it was an NHS project. After adding these icons, we found that the response from our TA and client was much more positive than previously – this made the UI feel more user friendly.

We also received a comment from our TA later on about the structure of the UI. We had the Add Voice Files button on the Generate IVR page. During our demonstration with the TA, he got quite confused about what this button was used for, he thought that it was another option to generate an IVR. After seeing his confusion, we realised that this was an issue, and it did not make sense to have this on the Generate IVR page when it was not a method to generate an IVR. We therefore moved this button to the Home page instead, hopefully clearing up confusion for users. We also renamed the button to Add Recorded Speech Files to IVR in order to make the use of this page much clearer to any users. Initially we also made the UI very small, the buttons were very thin on each page and everything was quite squished together. After our first iteration, we realised this, and we decided to enlarge everything and add a lot more space between widgets in order to improve visibility and usability for the user. Our client also wanted us to be able to run the system on a Linode server. A Linode server has a much smaller screen resolution and size. This meant that we had to carefully size our program, so that everything was very visible, but it still fit on a Linode server screen.

We also had feedback from our TA that it might be a good idea to add an Exit button so that the user knows it is safe to exit the application. We appreciated that this was a good idea, to make it clear that exiting the application would not turn off the IVR server. Although this is stressed in our user guide, we also have an Exit button. This can be accessed by clicking the Menu button in the top left of the screen from any page and clicking Exit. A user can also use the shortcut – CTRL+Q

Menu & Home Page

  • We use different colours for Building, Online or Offline IVR statuses. This is used as a visible indicator to clearly show what the state of the IVR server is to the User.
  • We also only make available the button/s that the user is allowed to click. If the IVR is online for example, the user is unable to click the Turn ON IVR Server button. This is so that the user knows exactly what their capabilities and options are at the time.

Server Status

  • We have error messages when a user does something incorrectly, such as typing in their Voiceflow credentials incorrectly or trying to add the wrong file type.
  • We also have a warning message when turning off the IVR because this would turn off the whole IVR system, making it quite a serious mistake if clicked accidentally.

    • Generate IVR

    We have landing page for the two different options to generate an IVR. We decided to make a whole section for this in order to simplify the UI for the user, separating this section from the Home page.

    When talking to our client early on, we were told that adding the capability to drop a file into the UI could simplify the process for the user. We therefore decided to add the capability for both dragging and dropping a file as well as clicking Browse in order to choose a file from the file explorer. This simplifies the process and makes the overall experience of choosing a file more user friendly.

    Add recorded Voicefile

    We realised with a couple of pages such as the page to add voice files that it would grow or shrink depending on the size of the call triage a user had created. Because we could not determine the size, so that the page didn’t expand off the screen for large IVR triages, we decided to add a scroll bar to some pages. This allows the user to easily view the entire contents of the page just by scrolling up or down.

    Statistical Analysis

    • When designing the Analytics page, we decided to use a different type of graph for each section of the analytics page. Although we went with the most appropriate format for each section, this makes the page more interesting and engaging to a user.
    • We also initially had no way to set specific dates and time periods over which to view the analytics data. When designing this feature, we decided to add a very familiar and user-friendly date picker in a calendar format (as seen below). This just made usage of this page a lot more simplistic and easier to grasp for users.
    • Using the matplotlib library for python, we were also able to add the ability for a user to easily manipulate and save the graph very easily from the toolbar above the graph.

    SIP Trunk Setting

    On the SIP Trunk Settings page, we realised that we could not guess how many Provider Contact IP Addresses the user may want to add. Because of this we’ve added a button Add IP Address and the page expands to fit all available IP Addresses.

  • As can be seen from the above images, the Menu button is available from all pages in the top left corner. This allows the user to easily navigate to all of the main pages in the system from any page they are on.
  • We also added a Back button to every page. This allows you to go to the previous page on the UI. This just means that if a user accidentally navigated to the wrong page or clicked the wrong button, they can easily go back to the previous page without renavigating to it from the Menu.
